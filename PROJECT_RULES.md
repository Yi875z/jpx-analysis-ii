## 0. AIの役割と権限境界

- AIの役割は「調査・設計・草案作成・分析・実装補助・確認画面の提示」まで。**最終承認は常に人間**が行う。
- 本番DB・価格・在庫・商品ページ・顧客メール・投資取引に対する無人更新は禁止（ルール6参照）。
- EC・投資・開発の3文脈を混同しないこと。本ファイルは開発文脈の規約である。

---

## 1. 固定ルール（10カ条）

1. **既存確認が先**: 実装前に既存ファイル・README・本ファイル・DBスキーマ・テスト・Git履歴を確認する。
2. **MVPファースト**: 最小構成で動くものを先に作り、段階的に拡張する。
3. **最小差分**: 既存動作を壊さない。変更は必要最小限の範囲に留める。
4. **秘密情報をチャットに出さない**: APIキー・パスワード・トークン・.env内容・Supabase Service Role Key の貼り付け禁止。
5. **仕様を推測しない**: DBカラム名・CSVフィールド・API仕様は推測せず、実物を確認するか人間に質問する。
6. **無人の本番更新禁止**: 本番DB・価格・在庫・商品ページ・顧客メール・投資取引は人間の承認なしに変更しない。
7. **実装後の報告**: 変更ファイル・実行コマンド・テスト結果・未解決事項・ロールバック方法を必ず報告する。
8. **細かいGitコミット**: 1つの論理的変更ごとにコミットする。
9. **ローカル/公開の分離**: ローカル専用情報と公開可能情報を分けて管理する（公開区分L0〜L3に従う）。
10. **不明点は「未確認」と明記**: 確認できなかった事項は推測で埋めず「未確認」と記載して残す。

## 2. 過去失敗チェックリスト（実装・レビュー時に必ず照合）

- [ ] **秘密情報の混入**: コミット前に Gitleaks 実行。新規リポジトリは Git 全履歴スキャンも実施。
- [ ] **DB設計の未確定着手**: PK・ユニーク制約・冪等性を先に確定。再投入は source_file 単位の DELETE→INSERT で行う。
- [ ] **OCR誤読の伝播**: raw列（読取生値）と calculated列（計算値）を必ず分離し、検証ステップを工程に含める。
- [ ] **曖昧ページの自動クロール**: 取得可否と保存・再利用可否を別々に確認してから実装する。
- [ ] **一気に作りすぎ**: 要件定義→既存確認→最小差分→細かいコミット→フェーズレビューの順を守る。
- [ ] **外部連携の全量実行**: 新しい連携（Drive等）は限定フォルダ・小規模での疎通テストを先に行う。

## 3. 3層分離アーキテクチャ（ロックイン回避の基本構造）

| 層 | 置くもの | 置き場所 | 寿命 |
|---|---|---|---|
| 第1層: 処理本体 | AIなしで動く Python / SQL / n8n | `src/`, `scripts/` | 不変（資産） |
| 第2層: 指示書 | 本ファイル・要件・手順のプレーン日本語 | `PROJECT_RULES.md`, `docs/` | 不変（資産） |
| 第3層: 薄皮 | Skills / Hooks / サブエージェント定義 | 各エージェントの設定箇所 | 使い捨て |

**規約:**

- 第3層には「第1層のスクリプトを呼ぶ」「第2層の指示書を読む」以外の処理ロジックを書かない。
- 第3層に処理ロジックを書きたくなったら、それは第1層にスクリプト化すべきサイン。
- エージェント固有の語彙（ツール名・機能名）を第2層の文書に持ち込まない。

## 4. タスク分解・受け渡し規約

- **単一動詞分解**: AIに渡す1タスク＝1動詞。「集めて整形してExcel化」ではなく「①抽出 ②分割 ③マッピング ④xlsx化」に割る。
- **ファイル受け渡し**: 工程間の受け渡しは会話ではなくファイルで行う。命名は `stepNN_内容.拡張子`（例: `step01_raw.json`）。
- **決定論テスト**: スキル化・自動化の前に「同じ入力で10回実行して10回同じ結果か」を問う。Noなら自動化せず、都度AIに委任する側に分類する。
- **1作業1検証**: 各工程に1行で判定できる成功基準を付ける（例: 「出力行数が4,000±50」）。
- **プロンプト調整の損切り**: 発動制御のために指示文を3回以上書き直したら設計ミスと判断し、タスクを分割し直すか人間トリガーに変更する。

## 5. プロジェクト固有情報

- **技術スタック**: Python / pandas / openpyxl / Supabase(PostgreSQL) / Streamlit（ダッシュボード localhost:8501）/ Claude API（Anthropic・レポート生成）/ n8n（週次自動化）
- **起動コマンド**:
  - ダッシュボード: `streamlit run dashboard/app.py` → http://localhost:8501
  - 週次レポート再生成: `python main.py --report-only --date YYYY-MM-DD`
  - 月次サマリー: `python main.py --monthly YYYY-MM` / Excel再生成: `python scripts/build_excel.py` / バックフィル: `python scripts/backfill_jpx.py --from YYYY-MM-DD`
- **テストコマンド**: なし（テストディレクトリ未整備・2026-06-10 時点。整備時は §5 を更新）
- **DBスキーマの正**: `db/schema.sql`（Supabase SQL Editor で実行。推測禁止）。主要テーブル `weekly_futures`（week_date / investor_type / futures_type / long_lots / short_lots / net_lots / index_close / net_amount_oku / source_url）
- **データソースと取得条件**: JPX先物CSV（`scripts/parse_futures_csv.py` v1.1）+ 現物XLS（`scripts/parse_spot_xls.py`）。投資家コードは下表で固定（誤ると全数値がズレる）:
  海外投資家=**60** / 信託銀行=**23** / 事業法人=**32** / 個人=30 / 自己=10
- **スケジュール実行**: n8n 毎週木曜20時に自動実行（api_server.py + Windowsスタートアップ登録）。Claude API でレポート生成。
- **環境差異**: GitHub Private remote = `jpx-analysis-ii`。秘密の実値は `config/.env`（`SUPABASE_URL` / `SUPABASE_KEY` サービスロール / `ANTHROPIC_API_KEY`）にのみ置き、ドキュメント・チャットに実値を書かない。
- **公開区分とその根拠**: L1 暫定（Private リポ・限定アクセス。投資分析のため一般公開しない。正式決定は未確認）
- **詳細な作業履歴・次タスク候補**: `HANDOVER_20260412.md`（バックエンド）/ `HANDOVER_DASHBOARD_20260413.md`（ダッシュボード）/ git 履歴を参照（本ファイルには進捗ログを書かない）。

### プロジェクト固有ルール

- **`C:\CarSol\JPX_Analysis_System` は別システム（証券会社別 建玉残高分析）。絶対に混同しない**。コードを相互流用しない。
  当プロジェクトは「投資主体別（外国人・個人・信託銀行等）の売買フロー」を扱う。
- **CSS設計原則**（ダッシュボード）: `[stMarkdownContainer] *` ワイルドカード禁止（KPI inline style を壊す）。
  KPIカード色は `.kpi-pos`/`.kpi-neg` クラスで管理。ポップオーバーは `body [data-baseweb="..."]`（portal描画）。Plotly色は `get_theme()` で動的切替。
- **Supabase の RLS を有効化**しておく（public スキーマ全テーブルで Row Level Security 有効・ポリシー不要。`FORCE` は付けない）。
- データが古い・範囲が短いときはサイドバー「キャッシュ更新」→ それでもダメなら Streamlit 再起動。1000行制限は `data_loader._fetch()` のページネーションで対応済み。

## 6. ハンドオフ規約（セッション間・エージェント間の引き継ぎ）

- セッション終了時、`docs/handoff/YYYYMMDD_handoff.md` に以下を残す:
  変更ファイル一覧 / 実行コマンド / テスト結果 / 未解決事項（「未確認」明記）/ ロールバック方法 / 次の一手。
- ハンドオフ文書はエージェント中立のプレーンMarkdownで書く（特定ツールの機能名に依存した記述を避ける）。
- 本ファイルの内容を変更した場合は、冒頭の「最終更新」を更新し、変更理由を1行で追記する。

## 7. 技術ノート（横断的な固定知識）

- **文字コード**: 日本の公的データはCP932が多い。読み込み時に `encoding` を明示し、推測に頼らない。
- **Pine Script**: v6をベースラインとする。`calc_bars_count` を明示。VIX先物はCBOE表記を使用。リペイント安全性と `request.security` 呼び出し上限に注意。
- **Supabase Free**: 一定期間未使用でプロジェクトが一時停止する。スケジュール実行前に稼働確認を入れる。
